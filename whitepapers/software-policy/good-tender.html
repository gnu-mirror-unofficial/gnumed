The 6 steps towards a good software tender

In order to achieve the goals of interoperability, reusability, and standard compliance the contract needs to be drafted accordingly. Of course, all common contract issues such as goal specifications, budget,default handling etc. need to be considered. This is usually no problem, since anybody used to drafting contracts will know how to do this.
However, in the case of publicly funded software, there are a few extra requirements that are essential as well: no contract should be put to tender which doesn't specify in detail all of the following issues:
  1. Which license for the end product?
    The license will ultimately determine the long term value of any software
  2. What platform shall be supported?
    It is unwise to restrict the choice of platform artificially without good reason.
  3. What programming languages and software tools can be used?
    The choice of development tools has a major impact on interoperability and reusability
  4. What API and/or RPC interface is to be used?
    Standardised APIs and RPC interfaces will increase interoperability and reusability dramatically.
  5. Which existing components are to be reused or extended?
    Reuse of existing components fosters standards and saves money
  6. What standards are to be used?
    The choice of standard is essential for interoperability
The rest of this document intends to clarify these six core issues. It will be highlighted why these issues are important, and advice will be given regarding relevant software contract specifications.
