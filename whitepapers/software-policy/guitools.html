Portable graphical user interface toolkits

Generally, they should
  • provide a native look & feel on each platform
  • interact with "native" applications in the expected way
  • not impact negatively on performance
  • be available for a variety of programming languages
A graphical user interface design tool is desirable. It is even more desirable if such design tool allows clean separation from user interface design and code, allowing to modify user interface details without having to adjust code manually.

Remarks
 Extremely robust and portable toolkit with highly praised support and documentation. Goes far beyond mere GUI portability, solving most cross platform portability problems. Excellent user interface designer / RAD tool. Possibly the most compact code of all GUI toolkits, usually less than 50% of the code necessary to achieve the same goal as compared to Microsoft's MFC or Motif. Decreases complexity of source code and thus increases development speed considerably as compared to most toolkits. Unfortunately, it is only free on Unix/X11, and then only if software developed with it is released under GPL license.
However, developer licenses are modestly priced and since no royalties are required, QT licenses are a worthwhile investment.

Conclusion portable GUI tools:
The most mature and versatile toolkit is probably QT. The two main drawbacks are that achieving "native look & feel" sometimes requires some tweaking (can be done by the installation procedure so that the end user needs not get confronted with this) and that a free GPL version is only available for Unix/X11. However, any budget that can afford a few thousand dollars for developer licenses, will not see any problem in this. Volunteers can simply develop on Linux using the free GPL version - since the QT code is identical on all platforms, this is no impediment.

Small utilities where size matters and limitation to C++ as language is no problem are probably best served by using FLTK. It does away with bloat, and its suitability for static linking allows production of single-file binaries without any external library dependencies - something appreciated by most users

Projects where the license restriction of QT matters and where native look & feel is absolutely crucial, are best served by using wxWindows or wxPython. This toolkit is robust and feature rich, and since it is just an API to native implementations on each platform it does not only look and feel "native", it is native. Similar to QT, wxWindows is entering the embedded market, making porting to handhelds easy.

The GTK toolkit, while one of the most used GUI toolkits used in the free software domain, unfortunately still lacks the professional Windows support that would yet be necessary to recommend it for true cross platform development

There is a vast choice of portable GUI tools available, both free and commercial, that have not been mentioned. A popular choice in the Windows world - MFC (Microsoft Foundation Classes)- has not been mentioned since it is entirely unportable. While excellent RAD tools exist for MFC, most of these tools make it difficult to separate GUI from application logic. They encourage and sometimes even enforce a sloppy and unportable programming style, effectively locking developers into a specific tool once complexity of a project has exceeded a certain limit. Thus, we would strongly advise against graphical user interfaces developed using MFC.

Another popular choice is TK, Tcl's toolkit. It is one of the most ported graphical user interface toolkits ever. Interfaces exist for most other languages as well, and currently it is still the "official standard" cross platform GUI interface for Python. It's main drawback is a feature poor standard widget set (if not using 3rd party extensions) as well as poor performance due to it's string based messaging and parameter architecture.

Java's "Swing" should probably have deserved a detailed review as well. It is feature rich, powerful and truly portable. Language bindings for languages other than Java exist, such as for Python. However, we are not the only ones getting negative feedback from users regarding slow performance and complete lack of native "look and feel". Since end user acceptance ranks very high on our priority list, the use of "Swing" should initially be reserved for applications that are mainly used by IT professionals who do not mind the lack of native "look and feel".
