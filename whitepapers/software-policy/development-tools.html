The choice of software tools

Software development often depends on specific software tools and third party software libraries.

Why it matters

The choice of tools and libraries strongly influences programming style, and thus maintainability and reusability of software.
Not all developers are equally comfortable with all tools. Being able to recruit suitable developers is often a matter of providing the right tools. Development speed can depend significantly on the choice of tools. Some vendors abuse software tools as a mechanism of implementing vendor lock-in even if the tools are used for open standard programming languages. These tools typically enforce tool-specific programming styles, making it very difficult to migrate to different tools for the same purpose (Example: MS Visual Studio).

Licensing and source code availability of third party software libraries is a most important issue that is commonly overlooked: the wrong license and unavailability of source code can render a whole project unreusable and unmaintainable unless significant further development costs are spent on replacing these third party libraries.

Learning from past mistakes

It is worth mentioning that two of the major Australian software projects are said to suffer difficulties with product adoption and project progress because of their choice of software tools and libraries:
  • The "Argus" project is using Borland's "JBuilder", a proprietary integrated development tool which encourages use of non-standard third party libraries (namely Borland's own). This has alienated them from volunteers from the free software community who could have helped to reduce development costs and accelerated the testing/debugging cycles significantly. Use of these libraries also results in lesser flexibility and doubtful long term maintainability.
  • The "GEHR" project has chosen to rely on proprietary extensions and class libraries to the Eiffel language - a language which itself is not used very often in the first place, despite it's doubtless merits. The initial implementation relied on a proprietary object database module ("POET") which made it impossible for the community to test the product and to get used to it that way. The costs associated with the chosen tools made it impossible for volunteers to join development - volunteers who would have been crucial for a chronically under funded project

General requirements for software tools and other third party software

It should be a general requirement that
  • no specific development tool should be allowed unless it is replaceable with at least one other software tool of which the long term availability is guaranteed. The latter applies obviously only to tools where the source code is available.
    Example: A developer may chose the proprietary "Microsoft Visual C++" as development tool as long as he produces ANSI compliant C++ code and portable makefiles
  • no third party code should be allowed unless it's license allows full transferable ownership and full availability of source code. Third party libraries should not compromise portability
  • before any decisions are made regarding software tools and libraries, extensive research on available free tools and libraries should be performed. Not knowing about existing time proven free tools and libraries has wasted fortunes (Example: HeSA's decision to purchase cryptographic modules despite availability of free fully tested cryptographic libraries such as OpenSSL)

Portability considerations

  • Without very good reason, code has to be written in a portable way
  • Whenever features require platform specific implementation, platform specific details have to be isolated via abstract interfaces so that porting can be achieved with least possible effort
  • Before such self written abstraction is taken into consideration, extensive research should be undertaken regarding available cross platform tool kits
  • Wherever possible, core functionality should be implemented without any graphical user interface
  • Wherever graphical user interfaces are necessary, the user interface part has to be cleanly separated from application logic
  • Graphical user interfaces have to be implemented with portable interface toolkits
  • These requirements are recursive: they apply to any tools and libraries used
Portability is usually affected by the following issues:
  • file system access
  • spawning and handling child / parallel processes
  • networking
  • inter-application interaction / remote procedure calls
  • user interface, both graphical or text mode
  • device interfacing (sound, 3D acceleration, printing etc)
  • end user software installation
Fortunately, most high level languages nowadays have standard libraries taking care of many of these issues. The general exception are graphical user interfaces - only Java appears to provide a consistent portable graphical user interface through it's standard libraries. However, this at the expense of platform-native "look & feel" - an issue many users dislike, but also an issue that leads to inconsistent behaviour on most platforms when interacting with non-Java applications. (Example: cut & paste handling, standard dialogues for file selection and error handling, short cuts). The other exception is device interfacing.

Available portable toolkits and third party libraries

The following list of available free software tools is a suggestion only, to serve as an example and starting point. As the list of recommended programming languages before, it is far from exhaustive but tries to cover the most common choices.

Essential criteria are:

  • availability on as many platforms as possible. At least covering Windows, Linux and Apple MacIntosh
  • ease of use
  • availability of quality documentation
  • royalty free distribution of developed software
  • source code availability to ensure long term maintainability and reusability
